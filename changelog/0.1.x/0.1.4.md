# 0.1.4 — 2026-06-10

## Fixed

- **AirNow cache keys** ([#9](https://github.com/cyanheads/epa-mcp-server/issues/9)) — keys no longer contain characters rejected by the storage-key validator (pattern `/^[a-zA-Z0-9_.\-/]+$/`). Previously every call crashed at the cache write because `kind` values like `current:zip` produced keys with colons (e.g. `airnow:current:zip:...`). Keys now use `/` and `.` separators with sanitized parameter values (e.g. `airnow/current/zip/distance-25_zipCode-98101`).
- **ECHO facility lookup** ([#10](https://github.com/cyanheads/epa-mcp-server/issues/10)) — `getFacility()` now calls `dfr_rest_services.get_dfr`, which accepts numeric FRS Registry IDs directly. The prior endpoint (`echo_rest_services.get_facility_info`) only accepts program-specific IDs (e.g. `WAD988470605`) and always returned 0 facilities for Registry IDs, causing every lookup to fail. Facility name and address are derived from the `Results.Permits[]` FRS record; facility identity confirmed via `Results.RegistryID`.
- **ECHO violations search** ([#11](https://github.com/cyanheads/epa-mcp-server/issues/11)) — `searchViolations()` now follows the correct two-step case flow: Step 1 calls `case_rest_services.get_case_info` to obtain `QueryID` and `QueryRows`; Step 2 fetches cases via `case_rest_services.get_qid?qid=<QueryID>`. Previously only Step 1 was used, which is a discovery endpoint that never returns a `Cases` array, so every search yielded `{ cases: [], totalCount: 0 }`. Real field names now mapped correctly: `CaseNumber`→`caseId`, `CaseCategoryDesc`→`caseType`, `PrimaryLaw`→`programsViolated`, `FedPenalty` (dollar string)→`penaltyAssessedInDollars` (float), `DateFiled`→`filedDate`. `FacName`, `RegistryID`, and `State` are absent from the `get_qid` response; output schema descriptions updated to reflect this.
